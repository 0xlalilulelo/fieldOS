# M3 — JIT Integration: Step 5 of the holyc-graft

*May 7 to May 8, 2026. Eleven commits across step 5's four sub-steps.*

M3-B's wrap closed with "Onward to step 5." Step 5 is the audit's
integration step: the kernel-resident subset has been compiling
HolyC source to AT&T text for a while, the in-tree encoder has
been emitting bytes byte-equivalent with GAS for a while, and the
JIT region has been clearing NX on demand for a while — but no
single call has yet driven all three at once. Step 5 builds the
seam.

[ADR-0001 §3 step 5](../adrs/0001-holyc-graft.md)'s exit gate is
verbatim: `holyc_eval("U0 F() { 'X\\n'; } F();")` prints `X` on
serial; boot-to-sentinel ≤ 3 s on TCG. Both met as of commit
`24f5447`. This devlog covers the four sub-steps that got there
plus the smoke-bracket commit that locks the witness in CI.

## What landed

Step 5 broke into four sub-candidates. Each one's commit body is
where the technical detail lives; this devlog is the connective
tissue.

### 5-1 — Encoder relocation API *(`03eebea`)*

The encoder up to M3-B encoded each line into bytes and lost the
information about which 4-byte field was a `rel32` to a label.
Step 5 needed that information back: the JIT-fill pass has to
patch local labels (whose offsets only become known after a full
walk) and defer extern symbols (whose virtual addresses come from
the ABI table). 5-1 added a `Reloc[]` out-parameter to
`asm_encode`: each instruction line that emits a `rel32` writes
one entry naming the symbol and the byte offset of the field
within the instruction. Bug_171.s surfaces nine such sites across
seven distinct symbols (`_FN`, `_PrintMessage`, `_printf` ×3,
`.L0`, `.L4`, etc.); the host harness asserts the count and the
distinct-symbol set on every CI run. The encoder's emit path
stayed unchanged byte-for-byte; the new information is purely
additive. Reloc capacity overflow returns `AS_E_NOSPACE` — the
same error class the existing `out_cap` overflow uses.

### 5-2 — The pipeline driver *(`2ad2f62`, `7bf9ed7`, `8a9db82`, `3314f3a`, `796bb55`)*

5-2 wires the four pieces — parser, codegen, encoder, JIT — into
a single `holyc_eval(const char *src)` call. The five sub-commits
each landed one observable forward step:

- **5-2a-prep** expanded the kernel-resident subset to reach
  `compileToAsm`. The M3-B subset had been linked sparingly (each
  commit pulled in only what its witness needed); to drive the
  full pipeline `eval.c` had to call into `cctrl.c`, `parser.c`,
  `compile.c`, `lexer.c`, and `aostr.c` simultaneously. The
  Make-rule fan-out was straightforward; the `--gc-sections`
  reachability worked in our favour as before.
- **5-2a** built the source-pointer-to-AoStr bridge that lets
  `holyc_eval` feed the lexer without a file path. Upstream's
  `compileToAst` calls `lexPushFile` with an open file descriptor;
  we call `lexInit` directly with the caller's string and
  synthesize a minimal `LexFile` so the error-reporting paths
  (`lexer.c:1322`, `lexer.c:1726`) have something to point at.
  Witness: `compileToAsm` returns AT&T text for the I64 form of
  the eventual exit witness, which the smoke output prints
  between two `Eval: compileToAsm ---` fences for inspection.
- **5-2b** is pass-1 of the line walker. Walks `compileToAsm`'s
  output line by line, calls `asm_encode` with `out=NULL` to get
  the byte count, and accumulates the running offset; lines
  ending `:` get recorded as labels paired with the offset they
  sit at. The label table is the source of truth for local rel32
  patches in pass-2 and for entry-point lookup in 5-4a.
- **5-2c** is pass-2. Re-walks the same buffer, this time
  emitting bytes into the JIT region and patching local rel32
  sites against the label table. Relocations whose symbol does
  not match a label append to an `HolycExternTable` for 5-3 to
  resolve. The patch math is `*(int32_t *)(out + off) =
  (int32_t)(target - (off + 4))`, evaluated at the kernel-side
  pointer (the JIT region is identity-mapped from the kernel's
  view).
- **5-2d** bridges the walker output into the JIT region: the
  buffer comes from `holyc_jit_alloc`, the bytes go in, then
  `holyc_jit_commit` flips NX off per page via `vmm_remap`. The
  witness line `Eval: pipeline... OK (N bytes, M relocs deferred)`
  lands on serial — at this point M was nonzero (printf and
  argc/argv unresolved), but the relocations existed and were
  surfaced cleanly. 5-3 closes the M=0 case.

### 5-3 — Extern resolution *(`a0e3b89`, `fc5063f`, `127de8a`, `621f4db`, `4be5db6`)*

5-3 is the ABI table and the pass that consumes it. ADR-0001 §3
step 5 had originally sketched a `k_*` shim TU sitting between
the kernel runtime and HolyC-resident code; M3-B's reading turned
that into a question — does each ABI entry need a shim, or can
the table point directly at existing kernel primitives?

The pick (HANDOFF.md trade-off pair, M3-B): direct-address
aliasing where the upstream emit's symbol name and the kernel's
existing primitive line up, with `k_*` shims landing only the
first time a HolyC source unit actually calls a `k_*` symbol. For
M3 step 5's witness this means `printf` resolves to `runtime.c`'s
existing `printf` (which itself wraps `serial_putc` for output);
no shim TU exists yet.

The five sub-commits:

- **5-3a** built `abi_table.c`: a small static table mapping
  symbol names to addresses, plus storage entries for `argc` and
  `argv`. The upstream emit unconditionally writes `rdi`/`rsi`
  into `argc(%rip)` / `argv(%rip)` in main's prologue regardless
  of whether the source uses them; the storage must exist or the
  rel32 lands in unmapped memory. 8-byte static `U64`s in
  `abi_table.c` absorb the writes silently.
- **5-3b** added function entries for `printf` and `_printf`
  (the Mach-O / ELF naming-prefix dual). Bug_171.s on the host
  exercises the prefixed form; the kernel hcc build (cross-GCC
  ELF) emits the unprefixed form. Both have to resolve. The
  commit also added a runtime canary asserting that the
  resolved address is non-zero, so an alias-drift regression
  fails the boot rather than presenting as a silent rel32 patch
  pointing at zero.
- **5-3c** is pass-3 of the walker. Walks the deferred extern
  table, calls `abi_table_lookup` for each entry, patches the
  rel32 against the resolved address. The patch math is the same
  as pass-2's, but the `target` comes from the lookup. Pass-3
  runs *before* `holyc_jit_commit` so the buffer is still
  writable irrespective of any future strict-W^X ADR. The
  `lookup` function pointer (rather than a direct call to
  `abi_table_lookup`) is what lets `walker.c` stay host-buildable
  in the asm-test harness without dragging the kernel-only ABI
  TU into the host link.
- **5-3d** is the unresolved policy: ADR-0001 §3 step 5 says
  `holyc_eval` refuses to run a module whose externs do not all
  resolve. Per-entry log on stderr-equivalent (serial), then
  `return -1`. The runtime witness for the unresolved branch was
  unreachable from a kernel-side test — the upstream parser
  rejects undeclared calls during `parseToAst` and exits via the
  runtime exit() shim before pass-3 sees them — so the witness
  lives in the host harness instead.
- **5-3e** is the host harness for pass-3 verification. Two
  mock lookups: one returning a fixed virtual address for
  `_printf` (asserts the rel32 displacement gets patched
  correctly with non-zero displacement); one returning zero
  unconditionally (asserts the unresolved-count path with all
  three deferred symbols still unresolved). Both run in CI.

### 5-4 — JIT invocation *(`22e42b0`, `dc4b26f`, `aba49dc`, `24f5447`, `f12d4d8`)*

5-4 is the close. The bytes are correct, the relocations are
patched, the page is W+X. What remained: take a function pointer
to `(out_buf + entry_offset)`, invoke it, and have the witness
print `X`.

- **5-4a** exposed pass-2's static `label_lookup` as a public
  `holyc_label_lookup`. The call site in `eval.c` looks up `main`
  immediately after pass-3 logging and prints `Eval: entry main
  @ N` so the upstream's wrapper shape is observable in the
  smoke. Lookup-by-name `main` (rather than the user's source
  function `F` or "first non-zero offset") is what the upstream
  emit guarantees regardless of source shape — the upstream
  always wraps the user's source unit in a SysV-prologued `main`
  that calls user code from inside.
- **5-4b** is the call site. After `holyc_jit_commit`, cast
  `(out_buf + entry_offset)` to `int (*)(int, char **)` and
  invoke with `(0, NULL)`. The return value lands on serial as
  `Eval: invoke main(0, NULL) -> rc=N`. With the I64 witness
  still in place at this commit, `N` was 42 — F's return value
  threading through `call F; leave; ret` and out through `%rax`
  to the C caller. ADR-0002 §1's IF=0 invariant survives the
  jump (kernel and JIT-resident code share ring 0; no new
  exception path opens); the M4 `fxsave` / `fxrstor` obligation
  in ADR-0002 §2 does not move forward because the witness has
  no float / xmm touch.
- **5-4c-prep** was the surprise. The witness swap to `'X\n'`
  triggered `AS_E_MALFORMED` in pass-1: `compileToAsm`'s
  `.asciz "%S"` emit writes the AoStr's raw bytes verbatim, so
  the HolyC source string `'X\n'` (parsed as `0x58 0x0A`) lands
  in the AT&T text as a literal newline byte *inside* the quoted
  operand. The line walker split on it, the encoder saw an
  unterminated quote, and pass-1 failed. The fix is a quote-
  aware line iterator: `\n` inside `"..."` is part of the same
  logical line. The encoder needed no changes —
  `parseStringOperand` already passes a literal control byte
  through to the output. Bug_171.s's checked-in corpus uses the
  escaped two-char form `\n` inside its `.asciz` lines so the
  asm-test path never enters the new branch and corpus parity
  stayed byte-equivalent. (ADR-0003 §1 forbids modifying
  `x86.c`, so the fix landed in the consumer.)
- **5-4c** is the witness swap. With the quote-aware walker in
  place, `holyc_eval("U0 F() { 'X\n'; } F();")` produces the
  right shape: 68 bytes, 6 labels (`.L0` for the string pool now
  joins the set), pass-2 patches 2 local rel32s and defers 3
  extern relocations (`printf`, `argc`, `argv`); pass-3 resolves
  all three. Commit + invoke + return: `X` lands between the
  entry log and the invoke log on serial, and `rc=2` reports
  printf's char-count return propagating through F → main's
  `%rax`. ADR-0001 §3 step 5 closes here.
- **5-4d** locks the witness in CI. `ci/qemu-smoke.sh` had been
  asserting `FIELD_OS_BOOT_OK` only; an awk bracket now also
  asserts the literal `X` line lands between `Eval: entry main`
  and `Eval: invoke main` in the serial log. Three distinct exit
  codes inside awk distinguish "missing entry log" (1), "missing
  invoke log" (2), and "no `X`-only line between them" (3); the
  outer script wraps any of those into smoke exit code 5
  (`jit_witness`). A jump to a wrong offset, a missed printf
  rel32 patch, or a swapped main/F call ordering would now fail
  CI rather than just look weird on serial.

## What surprised us

5-4c was supposed to be a 5-LOC swap. The HANDOFF.md kickoff
explicitly flagged the lowering as the unverified assumption —
that text was the right caution and the right guess about *which
direction* the surprise would come from (a missing ABI entry was
the predicted shape; the actual shape was a missing line walker
mode). That the surprise hit at all is unremarkable. That the
fix was small and surgical — twenty-nine lines in `walker.c`,
with the encoder unchanged — is what step 5's discipline made
possible. The single-language commitment plus the corpus-driven
encoder discipline plus ADR-0003's "do not modify `x86.c`" plus
the table-driven encoder structure all compounded to keep the
blast radius small.

The other thing worth recording: the rc=2 result was unplanned
diagnostic value. The U0 return type means the witness's return
is informational, but the SysV ABI returns `%rax` regardless,
and printf's char-count return propagating through F → main and
back to the C caller is a strong "the call entered, the printf
fired, control returned" signal. Without 5-4b's intermediate
`Eval: invoke main(0, NULL) -> rc=N` log we wouldn't see it. The
HANDOFF.md trade-off pair "5-4b + 5-4c combined?" — rejected at
the kickoff for losing the rc==42 intermediate signal — was the
right call for a different reason than originally argued.

## Cadence

Eleven commits across May 7 and May 8. Two evening sessions plus
one weekend morning. LOC budget: 5,921 / 100,000 (5.9 %), 52
base-system files. The line delta for step 5 is +112 (5-1's API
landed in M3-B's count) plus the host-harness work (excluded
from the budget). The smoke test is green. Boot-to-sentinel on
TCG: ~2 s including the new JIT invocation.

The encoder's coverage on `Bug_171.s`: 63 / 63 instruction
lines, 6 / 6 directive-data lines, 9 relocations across 7
distinct symbols, 11 labels @ 227 bytes, pass-2 patches 6 local
rel32s and defers 3 externs (`_printf` ×3), pass-3 resolves 3
against the mock and 0 against the empty-mock. Kernel-side, the
JIT invocation produces 68 bytes with 6 labels (`sign_bit`,
`one_dbl`, `.L0`, `F`, `main`, `.LFE0`), 2 local patches, 3
extern resolutions, and `X` on serial.

## What remains in M3

One step. Step 6 is the REPL: `kernel/holyc/repl.c` reads lines
from `serial_getc` (the polled input that landed in M3-A) and
dispatches each line to `holyc_eval`. Five-line scrollback
minimum. Exit gate (`phase-0.md` §M3, verbatim): a five-line
REPL session over serial covering arithmetic, function
definition, function call, variable mutation, and a deliberate
parse error that does not crash the kernel.

Step 6 is mostly serial I/O on top of an already working
compiler. The compiler is now observably working: it accepts
source, walks it through parse + codegen + encode + relocate +
commit + invoke, and the emitted code prints to serial. The
question step 6 asks is "what does the line buffer look like,
and how does parse-error recovery work without crashing the
kernel" — neither of which is two-weeks-shaped. The Path-B
trigger from ADR-0001 §4 stays nominal.

The plan is holding. Onward to step 6.
