# M2 — Memory Management

*May 5 to May 6, 2026. Four commits.*

M2 is where the kernel learns to allocate. M0 was tooling and a
bootloader handshake; M1 added GDT, IDT, and a framebuffer to
prove the kernel could survive contact with the CPU. M2 is the
first milestone where the kernel has its own memory model. There
is no public-facing change at all — the screen still says
`Hello, Field` and nothing else — but on serial three new lines
of self-test now print every boot and certify that the PMM, VMM,
and slab allocator are not silently broken.

## What landed

Four commits, in order:

- `40cc270` *feat(mm): bitmap physical-memory manager.* One bit
  per 4 KiB frame in a packed `uint8_t` array. `pmm_init` walks
  the Limine memmap, sizes the bitmap from the highest USABLE
  address, and bootstraps the bitmap pages inside the largest
  USABLE region — they self-mark as used in their first scan.
  The free-frame search is a two-finger cursor over `uint64_t`
  words using `__builtin_ctzll` for the bit-scan. No buddy yet;
  Phase 0 explicitly defers it until profiling demands it. Smoke
  prints `Memory: 254 MiB free of 254 MiB total` under
  `qemu -m 256M` — the 2 MiB shortfall is the firmware/Limine
  reservation visible in the memmap.
- `06bee00` *feat(mm): 4-level VMM with map/unmap/translate;
  1 GiB self-test.* `vmm_map(pml4, virt, phys, flags)`,
  `vmm_unmap`, `vmm_translate` over an arbitrary PML4. Walker
  reaches page-table memory through the Limine HHDM rather than
  through recursive mapping — the recursive trick is clever and
  it bites later under SMP, per phase-0.md §M2 sub-goal 2.
  `vmm_new_address_space` clones entries 256..511 from a master
  PML4 captured from CR3 at init, so process-private lower-half
  spaces inherit kernel mappings without sharing the lower half.
  Self-test runs every boot: 256K iterations of map / translate /
  unmap over 1 GiB of virtual addresses based at 16 TiB. Output:
  `OK (PMM retained 513 pages = 2052 KiB for page tables)` —
  exactly one PD plus 512 PTs, the predicted overhead. Reclaim
  of intermediate page tables is deferred; bounded and acceptable
  for now.
- `5fbc074` *feat(mm): slab heap with 8 caches and contiguous-page
  large path.* Eight per-size caches at 16, 32, 64, 128, 256,
  512, 1024, 2048 bytes over 4 KiB PMM pages. Each slab page
  carries a 32-byte header at offset 0 — magic 0x5A1B,
  cache_id, free_count, total_slots, first_free, prev/next list
  pointers. Slots are packed after the header for sizes ≤ 512;
  for the 1024 and 2048 caches the first slot starts at offset
  `cache_size` so the slot grid stays size-aligned. Each free
  slot's first 2 bytes thread the freelist as a `uint16_t` page
  offset, which is enough because the whole slab fits in one
  page. Allocations larger than 2 KiB take a contiguous N-page
  run from the PMM with a 16-byte header (magic 0x1A1B, page
  count) at offset 0 and payload at offset 16. `kfree(ptr)`
  masks to the 4 KiB page boundary, reads the magic word,
  dispatches to slab-free or large-free; bad magic panics via
  `cli; hlt`. Fully-empty slab pages unlink from their cache
  list and return their page to the PMM — the no-leaks
  self-test fails without it.
- `6cd9855` *feat(mm): M2 wrap; format helper; stage 2.* Bumped
  the serial stage marker from `stage 1 reached` to `stage 2
  reached`, landing after every M2 init returns successfully.
  Consolidated four near-identical decimal printers
  (`idt.c::put_dec`, three `serial_print_dec` copies in
  `pmm.c` / `vmm.c` / `slab.c`) into a single
  `kernel/lib/format.{h,c}::format_dec(uint64_t)`. Net −34 LOC
  and one new directory. `idt.c::put_hex64` stays put — single
  panic-path caller, moves the next time a non-panic caller
  needs hex.

## The numbers

- **4 commits.**
- **+1,814 LOC of base-system code over M2.** PMM ≈ 250 LOC,
  VMM ≈ 280 LOC, slab ≈ 460 LOC, supporting changes scattered.
  Total kernel C is now **1,980 / 100,000** (2 %), 23 files.
- **3 new self-test lines on serial every boot.** Each one is a
  smoke gate; CI greps for `FIELD_OS_BOOT_OK` and any failure
  short-circuits before the sentinel.
- **513 pages of page-table overhead** retained after the VMM
  self-test (= 2052 KiB), exactly the predicted PD + 512 PT
  count for the 1 GiB working set.
- **~2 seconds** from QEMU launch to `FIELD_OS_BOOT_OK` in
  headless TCG. The 10K-iteration slab self-test is most of the
  delta from M1.

## What the boot looks like now

Serial:
```
Memory: 254 MiB free of 254 MiB total
VMM: 1 GiB map/unmap... OK (PMM retained 513 pages = 2052 KiB for page tables)
Slab: 10K random alloc/free... OK (no leaks)
Field OS: stage 2 reached
FIELD_OS_BOOT_OK
```

Framebuffer:
```
Hello, Field
```

The framebuffer is unchanged from M1. The four lines before the
sentinel are M2's gift — every boot of every commit going
forward is a memory-management regression test.

## What M3 looks like

M3 is the Big Risk: a HolyC compiler inside the kernel that
takes source over serial, compiles to x86_64 in memory,
executes, and prints. Bootstrap stays C — the C kernel calls
`holyc_init()` and `holyc_repl()` after memory and serial are
up. Phase-0.md flags it at 3–6 FT-weeks (~7–14 weeks
part-time). I do not expect it to compress.

## Cadence

Thirteen commits across the project. Three milestones complete
of about 130. M0 and M1 compressed; M2 compressed less but still
came in under the plan's calibration. M3 is where compression
stops being likely. The Big Risk milestone is named that
because the failure mode is "stall for two weeks on a parser
bug and lose momentum" — and the answer to that is the
C-bootstrap fallback, which keeps the kernel bootable and
shippable even if the HolyC graft slips.

The plan is holding. The line-count budget is at 2 %. The smoke
test is green and has been green every commit since M0 step 3.
Onward.
