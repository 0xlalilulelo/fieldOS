# M0 — Tooling and Bootstrap

*April 29 to April 30, 2026. Two evening sessions. Six commits.*

This is the first devlog for Field OS. M0 was the bookkeeping
milestone: get a cross-compiler, a Limine-bootable kernel skeleton,
a CI smoke loop, the holyc-lang fork vendored, and a boot signature
on serial. No HolyC runs yet, no JIT, no compositor. Those start in
M3 and M6. M0's exit criterion is "the entire chain compiles and
boots a do-nothing kernel that prints a known string." That criterion
is now met.

## What landed

Six commits, in order:

- `4d1e77a` *chore(repo): scaffold directory layout, license, naming
  catalog.* Repo skeleton, BSD-2 LICENSE, README, naming catalog
  lifted from CLAUDE.md to `docs/naming.md`, ADR template at
  `docs/adrs/0000-template.md`.

  `tools/build-toolchain.sh` is an idempotent builder for binutils

  canonical higher-half virtual base `0xffffffff80000000`.

  Reproducibility job deferred to M10.
- `60e1a48` *feat(holyc): vendor holyc-lang beta-v0.0.10 with
  audit.* ~21k LOC of upstream HolyC compiler at `holyc/` (BSD-2)
  plus a 200-line audit at `docs/skills/holyc-lang-audit.md`
  documenting architecture, libc surface, ABI, and a six-step M3
  graft roadmap. No vendored sources are modified in M0.

## How long it took

Two evening sessions on Apple Silicon, 2026-04-29 and 2026-04-30,
on the order of five hours of active time. Phase-0.md's calibrated
band was 5 to 7 part-time weeks. The work was faster than calibrated
for two reasons that don't generalise: the toolchain script and CI
YAML are well-trodden ground, and the holyc-lang audit is a reading
pass rather than a refactor. Phase-0.md is explicit that the
modification work on the compiler is M3.

The faster-than-calibrated read isn't a forecast about M1 onwards.
M3 has the freestanding-backend graft, which is real engineering.
The 3 to 6 FT-week estimate for that is the calibration I trust most
in the plan today.

## Detours worth recording

Three things broke that the OSDev wiki and Limine docs didn't warn
about clearly. Recording them so future-me doesn't lose another
half hour the next time I bring this up on a fresh host.

**binutils 2.42's bundled zlib won't build against macOS SDK 14+.**
The bundled zlib uses K&R-style function declarations, and
`<stdio.h>` on recent macOS SDKs has nullability annotations that
the compiler can't reconcile with K&R prototypes. Fix: pass
`--with-system-zlib` to both binutils and gcc configure. Benign on
Linux, where the system zlib is universally available. Cost: ~30
minutes staring at the actual compiler error before recognising it.

**`brew --prefix <pkg>` returns success even when the package is
not installed.** It returns the *prospective* install path for any
known formula. My initial probe slipped past a missing trio of
gmp/mpfr/libmpc, which then blew up GCC's configure with a less
helpful error from a different layer. Switched to
`brew ls --versions` for the existence check, which only succeeds
when actually installed. Cost: ~15 minutes.

**HVF cannot virtualise foreign-arch guests.** I had written
`tools/qemu-run.sh` to select `-accel hvf` on macOS unconditionally.
HVF only applies when host CPU arch matches guest arch. On Apple
Silicon (aarch64) running an x86_64 guest, HVF refuses; we fall
back to TCG. The script now keys on `uname -s`-`uname -m`. Cost:
the time to read the QEMU error and rewrite two case branches.

**macOS .dSYM bundles snuck into a vendor commit.** When `make -C
vendor/limine` builds the host limine tool with `cc -g`, clang
emits a `limine.dSYM/` directory next to the binary — debug
symbols. I had ignored the binary itself but not the bundle. Caught
it in `git log` after the fact and amended the commit; the only
amend in the M0 history.

None of these are M0-shaped problems. They are "first time I run
this exact sequence on this exact host" problems, the kind of
first-time-doing-this tax phase-0.md's risk register §6 warns about
without naming.

## The numbers

- **6 commits.** Each ships a self-contained piece of the M0 spec.
  Bisect surface is clean.
- **166 lines of base-system C** (`kernel/main.c` + the four files
  in `kernel/arch/x86_64/`). 0.17 % of the 100,000-line budget.
- **~17 MB of vendored Limine** (BSD-2 / 0BSD), 16 MB of which is
  the upstream UEFI El Torito boot image — a fixed-size FAT
  container that we can't reasonably shrink.
- **~35 MB of vendored holyc-lang** (BSD-2). Untouched until M3.
- **2 seconds** wall time from QEMU launch to `FIELD_OS_BOOT_OK`
  on serial in headless TCG. The kernel itself is ~17 KB ELF, most
  of which is `.debug_*`.

## What the boot looks like

The serial output is the entire M0 user-facing surface, and it is
two lines:

```
Field OS: stage 0 reached
FIELD_OS_BOOT_OK
```

The first line is the human stage indicator that scales by milestone
(M1 prints `stage 1 reached`, M2 prints `stage 2 reached`, etc.).
The second is the machine-readable sentinel that CI greps. There is
no compositor, no framebuffer write, no input loop, no scheduler.
The kernel initialises COM1 at 115200 8N1, prints its two lines,
and `cli; hlt`s forever.

That is, deliberately, all M0 promises.

## What M1 looks like

M1 is the first milestone where the kernel does something beyond
"I am here." Per phase-0.md §M1, the work:

- GDT with 5 entries (null, kernel code, kernel data, user code,
  user data) plus a TSS.
- IDT with 256 stub handlers, exceptions 0–31 routed to a shared
  panic path.
- IST stacks for `#DF`, `#NMI`, `#MC`, so the kernel can crash
  cleanly even from a broken state.
- Framebuffer-backed `Println` next to the existing serial path.
- A `Hello, Field` line on the framebuffer in 8×8 bitmap text.

Honest estimate: 1–2 FT-weeks per phase-0.md, ~3 weeks part-time at
15 h/wk. Interrupt and exception plumbing is the source of subtle
bugs that only surface much later. M1 should take longer in
calendar time than M0 did.
