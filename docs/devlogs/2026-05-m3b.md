# M3-B — The Kernel-Resident Subset and the In-Tree Encoder

*May 6 to May 7, 2026. Seventeen commits across two of M3's six
steps.*

M3 is the Big Risk. Phase 0 calls it that because the failure mode
is "stall for two weeks on a parser bug and lose momentum," and
because the work is the highest-novelty in the plan: a fork of
[holyc-lang](https://github.com/Jamesbarford/holyc-lang) running
inside the kernel, accepting source over serial, compiling to
x86_64 in memory, executing, and printing the result.
`docs/skills/holyc-lang-audit.md` decomposes the graft into six
steps. M3-A — the ground-prep half: the JIT region with per-page
NX flips, polled serial input on COM1, the host build wired into
the cross-compiled tree, ADR-0001 turning the audit's roadmap into
go/no-go gates, and the C-to-HolyC ABI header — landed first.
M3-B is steps 3 and 4: the kernel-resident subset and the in-tree
x86_64 encoder.

This devlog wraps M3-B. Steps 5 and 6 — JIT integration and the
REPL — are unblocked but unlanded; M3 is not yet complete.

## What landed

### Step 3 — Kernel-resident subset *(eight commits)*

The audit's step 3 strips host assumptions from the vendored
compiler. `<stdio.h>`, `<unistd.h>`, `<fcntl.h>`, the three
`system()` calls in `main.c`, the `cmake`-driven build — none of
these survive in the kernel image. ADR-0001 §3 step 3 names the
files removed (`transpiler.c`, `cfg*.c`, `cli.c`, `main.c`,
`memory.c`, `mempool.c`) and the files added (`runtime.c` for the
freestanding allocator and string shim; `eval.c` for the
`holyc_eval(const char *src)` entry).

The arc in commit order: a freestanding runtime providing the
small libc surface upstream uses (`memcpy`, `memmove`, `memset`,
`strlen`, `strcmp`, `strchr`, `strdup`, a serial-backed `printf`)
plus a weak-symbol allocator shim that points upstream's
`xmalloc` / `xfree` at the kernel slab; a Make rule set under
`kernel/kernel.mk` that compiles the trimmed source list with
`$(CROSS_CC) -ffreestanding -mno-sse -mno-sse2 -mcmodel=kernel`;
a five-gap journey through the runtime where each commit's witness
was a single upstream function the kernel build could now resolve
(aostr → ast → containers → realloc); `holyc_eval` skeleton wired
into `kmain` after `slab_init`; a partial-link expansion to
`ast.o` / `arena.o` / `containers.o`; the full subset linked into
the kernel ELF with the aoStr probe as a runtime witness; and
finally `cctrlNew` reaching real upstream code on serial — the
exit gate for step 3.

Through all of it the smoke test stayed green. `--gc-sections`
strips the kernel-resident subset from the ELF until something in
`kmain`'s reachability set calls into it, so each commit added
source without changing the boot image. The witnesses were probe
calls in `kmain` that touched one upstream function at a time and
printed a serial line confirming the linker found a definition,
the function ran, and returned without panic. The probes have been
factored back out at step 3 exit; the live witnesses today are the
two lines `Subset: hcc beta-v0.0.10 ready... OK` and
`Eval: skeleton... OK` printed every boot.

### Step 4 — In-tree x86_64 encoder *(nine commits)*

Step 4 is the audit's open question: how does the kernel get from
holyc-lang's AT&T-syntax assembly text to executable bytes, given
that upstream shells out to host `gcc`? ADR-0001 §2 closed three
options: vendoring keystone-engine (GPLv2, incompatible with the
BSD-2 base), vendoring asmjit (BSD/Zlib but C++, conflicts with
the single-language ethos), and modifying `x86.c` to emit binary
directly (a deep fork of upstream's 2,460-line codegen). The chosen
path was an in-tree, table-driven encoder consuming the AT&T text
upstream already produces. The
[OSDev x86-64 instruction encoding page](https://wiki.osdev.org/X86-64_Instruction_Encoding)
and Intel SDM Vol. 2 are the references; the encoder lives at
`kernel/holyc/asm.c`.

The audit estimated ~1,500 lines for this work. The actual landing
covering the full Bug_171.s surface is 926 LOC of `asm.c` plus
310 LOC of host harness (excluded from the budget under the
`*_test.c` rule that landed in B); future corpus inputs will grow
both, but the audit's estimate held up well as a ceiling. The arc
sub-decomposed into nine commits, each with a measured witness:

- **F** *(`7c50596`)*. ADR-0003 supersedes ADR-0001 §3 step 4's
  "Files modified: holyc/src/x86.c" line. On reading `x86.c` end
  to end, the audit's presupposed `FILE *` sink turned out to be
  an `AoStr` already — `compileToAsm()` returns the asm text in
  memory. The redirect work the original paragraph described was
  unnecessary; the discovery work it pointed at — what AT&T forms
  `x86.c` actually emits — survived as a host-side corpus capture
  rather than as an in-place edit. ADR-0001 §1's pin discipline
  tightened: `x86.c` is now untouched, and the next pin re-eval
  pays no rebase cost.
- **A′** *(`29c071e`)*. `holyc/tools/dump-asm.c` drives the host
  hcc through `compileToAst` → `compileToAsm` and writes the
  resulting `AoStr` to a file. `make corpus` produces
  `holyc/tests/corpus/Bug_171.s` from `Bug_171.HC` — 108 lines,
  checked in. The corpus is the encoder's measured spec: 63
  instruction lines, 6 directive-data lines, the rest labels /
  comments / non-emitting directives / blanks.
- **B** *(`4166ce9`)*. Encoder interface (`asm_encode(att_line,
  len, out, cap, *out_len)` with five return codes) and a v0 stub
  that returns `AS_E_TODO` for everything; harness (`asm_test.c`,
  167 lines, host-only) that classifies each corpus line and
  reports counts by return-code bucket. `tools/count-loc.sh`
  gained a `! -name '*_test.c'` exclusion, establishing the
  host-only-test naming convention for the project.
- **C-1** *(`4a6ca20`)*. Encoder skeleton plus the zero-operand
  and single-register-operand cluster: `ret`, `leave`, `push`
  reg64, `pop` reg64. `asm_test.c` gained the `$(CROSS_AS)` /
  `$(CROSS_OBJCOPY)` byte-comparison plumbing that ADR-0001 §3
  step 4's exit gate requires — write the line standalone, run
  the cross assembler, extract `.text`, compare bytes. **33/63
  instruction lines** matched GAS byte-for-byte.
- **C-2** *(`5bee29c`)*. The operand-parsing infrastructure every
  subsequent cluster depends on: `OPR_IMM` / `OPR_MEM` /
  `OPR_MEM_RIP`, signed integer parser, memory-operand parser,
  the `emitMem` helper covering Intel SDM Vol. 2 §2.1.5's three
  ModR/M special cases (rbp/r13 with disp=0; rsp/r12 SIB
  requirement; RIP-relative). `movq` four forms, `leaq` two
  forms. **+22 → 55/63**.
- **C-3** *(`4a16f04`)*. The last two operand kinds (`OPR_SYMBOL`
  for jump/call targets, `OPR_INDIRECT` for `*%reg`); `subq`
  imm-reg, `test` reg-reg, `je` rel32, `call` (symbol and
  indirect). **+8 → 63/63 instruction lines green.**
- **D-1** *(`9b04a07`)*. The data-directive cluster opens.
  `isNonEmittingLine` learns to not swallow data directives;
  five new encoders (`.byte`, `.short` / `.word`, `.long`,
  `.quad`) share a width-parameterised core that re-uses the
  signed-integer parser. The harness fans out: `LINE_INST` and
  `LINE_DIRECTIVE_DATA` both go through the encoder + GAS
  comparison loop, with two-axis ok / unknown counters per kind
  so each D commit's progress reads cleanly. CI gains a sibling
  `asm-test` job alongside `build-iso` / `smoke` / `loc-budget`.
  **1/6 directive-data lines** byte-equivalent.
- **D-2** *(`dc85e21`)*. String directives. `parseStringOperand`
  with `\n \r \t \\ \" \0 \xHH` escape handling; `.ascii` (no
  terminator), `.asciz` and `.string` (string + NUL — synonymous
  on x86 GAS). **+4 → 5/6 directive-data lines.**
- **D-3** *(`903a3ea`)*. Float directives. `parseDecimalOperand`
  parses the GAS decimal-float grammar with integer arithmetic
  only — `asm.c` also compiles under the kernel's
  `-mno-mmx -mno-sse -mno-sse2`, and host-side FP at encode time
  would emit x87 instructions the kernel cannot run. The parser
  succeeds iff the literal denotes an exact non-negative integer
  after exp-folding; genuine fractions return `AS_E_UNKNOWN`
  (coverage gap, not silent miscompile — the discipline of
  ADR-0001 §2's risk note). The integer-to-IEEE-754 builder is
  exact for `|v| < 2^53` (binary64) and `< 2^24` (binary32).
  **+1 → 6/6.**

ADR-0001 §3 step 4's exit gate is closed. Both halves: 69
emitting lines (63 instructions + 6 directive-data) match
`$(CROSS_AS)` byte-for-byte; the harness runs in CI on every push
and PR.

## The numbers

- **17 commits** in M3-B. Eight in step 3, nine in step 4.
- **+2,693 LOC of base-system code** since M2 wrap (1,980 →
  4,673). Of that: ~926 LOC of `asm.c` (the in-tree encoder); the
  remainder is the kernel-resident hcc subset (`runtime.c`,
  `eval.c`, the upstream subset linked under `--gc-sections`)
  plus plumbing in `kernel.mk` and the abi header. Total kernel
  C is now **4,673 / 100,000 (4.7 %), 46 files.**
- **310 LOC of host-only test harness** in `asm_test.c`,
  excluded from the budget per the `*_test.c` rule.
- **Encoder coverage on Bug_171.s**: 69/69 emitting lines
  byte-equivalent to GAS. Zero mismatches, zero malformed, zero
  nospace, zero gas-failed, zero unknown.
- **CI now has four jobs**: `build-iso`, `smoke`, `asm-test`,
  `loc-budget`. The `asm-test` job shares the toolchain cache
  key with `build-iso`; warm-cache runs are ~30 s.
- **~2 seconds** from QEMU launch to `FIELD_OS_BOOT_OK` —
  unchanged from M2. None of the M3-B work runs on the boot path
  yet; `--gc-sections` strips the unreached encoder symbols from
  the kernel ELF until step 5 wires the JIT region to them.

## What remains in M3

Two steps. Step 5 is integration: `eval.c` calls
`holyc_jit_alloc(byte_count)` for the encoder's output buffer,
populates it from `compileToAsm()` parsed line by line through
`asm_encode`, then `holyc_jit_commit(addr, len)` to flip NX off
on the populated pages. Exit gate (ADR-0001 §3 step 5):
`holyc_eval("U0 F() { 'X\\n'; } F();")` prints `X` on serial.

Step 6 is the REPL: `kernel/holyc/repl.c` reads lines from
`serial_getc` (the polled input that landed in M3-A) and
dispatches each line to `holyc_eval`. Five-line scrollback
minimum. Exit gate (`phase-0.md` §M3, verbatim): five-line REPL
session over serial covering arithmetic, function definition,
function call, variable mutation, and a deliberate parse error
that does not crash the kernel.

Neither step adds new architectural questions. Step 5 is mostly
rewiring; step 6 is mostly serial I/O on top of an already
working compiler. The Path-B trigger from ADR-0001 §4 — switching
to the Python `holyc2c` transpiler if any single step accumulates
two calendar weeks without reaching its exit gate — applies to
both, but neither feels two-weeks-shaped.

## Cadence

Forty commits across the project. Three milestones complete of
roughly 130; M3 partly complete (steps 1–4 of 6) and unblocked
toward its exit.
The line-count budget is at 4.7 %. The smoke test is green and
has been green every commit since M0 step 3.

Two notes worth keeping. First: the corpus-driven encoder
discipline worked. Specifying the encoder against measured AT&T
output rather than against synthetic input meant every commit's
witness was concrete — N lines green, M lines unknown — and the
unknown bucket surfaced coverage gaps without ever silently
miscompiling. ADR-0003 was the cheapest paper this project has
written; the work it shaped came in well under the audit's LOC
estimate (926 against ~1,500), with room to grow as future
corpus inputs add forms.

Second: the `*_test.c` budget exclusion landed at the right time.
The 310-line host harness would have been a non-trivial slice of
the encoder's LOC if it counted; instead it lives outside the
budget alongside the corpus, and the convention is set for
kernel-side test harnesses to come. The discipline is not
"tests are free" — the discipline is "test code that runs only
on the developer host does not eat the kernel-on-disk budget."
`tools/count-loc.sh` is the single point of truth.

The plan is holding. Onward to step 5.
