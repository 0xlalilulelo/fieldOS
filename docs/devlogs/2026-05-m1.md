# M1 ŌĆö Boot to Long Mode

*April 30 to early May 2026. Four commits.*

M1 is the first milestone where the kernel does something beyond
"I am here." M0 ended with a 17 KB ELF that printed two lines on
serial through Limine's hand-off and halted. M1 ended with a
kernel that owns its own GDT, can take an exception cleanly, has a
framebuffer, and renders text in a real bitmap font. The
public-facing change is a single new line on screen ŌĆö `Hello,
Field` in white-on-black 8├Ś8 ŌĆö but everything underneath is real
kernel plumbing.

## What landed

Four commits, in order:

- `5cda611` *feat(arch/x86_64): GDT with kernel/user selectors and
  TSS.* A 5-entry GDT (null + kernel/user code/data) plus a
  16-byte TSS descriptor. `gdt_init()` builds the descriptors,
  `lgdt`s the table, and reloads CS through a far-return trick
  while reloading SS/DS/ES/FS/GS to the kernel data selector.
  `ltr` installs the TSS into TR. The GDT load is observable
  only by survival ŌĆö a malformed descriptor would triple-fault
  before the sentinel ever printed.
- `5e78b1d` *feat(arch/x86_64): IDT with 256 stubs, IST, panic
  plumbing.* A 256-entry IDT with all gates pointing at one of
  256 per-vector asm stubs generated by a `.altmacro / .rept`
  pair. The stubs push a synthetic zero error code for vectors
  that the CPU does not push one for, then push the vector
  number, then jump to a common dispatcher that saves all 15
  GPRs into a `struct regs` and calls a C handler. The handler
  prints `PANIC: vec=N err=0x... rip=0x... rsp=0x...` over
  serial and halts. IST stacks (4 KiB each) for `#DF`, `#NMI`,
  and `#MC` so the kernel can crash cleanly even from a wrecked
  context. Verified by inserting a temporary `ud2` and watching
  `PANIC: vec=6` land on serial.
- `3171560` *feat(arch/x86_64): Limine framebuffer + 8├Ś8 font;
  "Hello, Field".* Limine framebuffer request, software glyph
  blitter, 256-glyph TempleOS 8├Ś8 console font copied verbatim
  from ZealOS `src/Kernel/FontStd.ZC` (Unlicense, public domain
  dedication ŌĆö lineage record kept per CLAUDE.md hard
  constraint #6). `fb_init` reads the response, captures
  pointer/pitch/dimensions, clears to black; `fb_putc_at` blits
  one 8├Ś8 cell against a 32 bpp BGRA framebuffer; `fb_puts`
  advances a cursor and wraps on `\n`. Visual eyeball plus a
  QEMU monitor screendump confirmed the glyph shapes.
- `c211cf8` *feat(kernel): finalize kmain order; M1 complete.*
  Switched the stage indicator from `stage 0` to `stage 1` and
  moved it after all M1 init, so its presence on serial
  certifies that every step succeeded. `qemu -d int,cpu_reset`
  showed zero exception or interrupt deliveries from the
  kernel; the only `CPU Reset` events in the log were
  firmware-side ŌĆö initial reset and a SeaBIOS SMM excursion,
  both still in real mode.

## How long it took

Four calendar days, evening sessions. Phase-0.md's calibrated
band for M1 is 1ŌĆō2 FT-weeks, so ~3 weeks part-time. The actual
work was tighter than that. Two reasons it compressed:

1. M0 step 3 had already done the boot-to-long-mode work via
   Limine. M1 explicitly only needed GDT, IDT, and framebuffer
   on top of an already-running kernel. The "boot to long mode"
   half of the milestone name was already paid for.
2. Limine's protocol does the heavy lifting. The kernel inherits
   a higher-half mapping, an HHDM, a usable memory map, and a
   framebuffer for free. Pre-Limine OS-dev work would have spent
   most of M1 on multiboot trampoline, A20 line, mode switching,
   and BIOS VBE shenanigans.

The compression is not a forecast for M2. M2 is the memory
manager, and that's real engineering ŌĆö page-table walker, slab
allocator, deciding what gets mapped where, surviving 10,000
random-sized allocations without leaking. Phase-0.md flags M2 as
"where most hobby OS projects either get it right and accelerate,
or get it wrong and bog down for months." I trust that
calibration more than M1's.

## Detours worth recording

Less drama than M0. The pieces that did bite:

**Aligning the SysV ABI before calling C from asm.** The common
exception dispatcher saves 15 GPRs and the synthetic
(vector, error_code) pair, then calls `exception_handler`. By
the time we reach the call, the CPU has pushed the exception
frame plus our pushes ŌĆö 22 qwords, 176 bytes. That is naturally
16-byte aligned. But x86_64 exception entry does not actually
guarantee 16-byte alignment for same-CPL non-IST entries; only
the IST ones get a fresh aligned stack. Solved cheaply with
`and $-16, %rsp` immediately before the `call`. Costs nothing,
makes the code correct under all delivery paths.

**HVF on Apple Silicon, again.** `tools/qemu-run.sh` had been
patched in M0 to fall through to TCG on aarch64 hosts, but the
GUI run (which I trust the user to eyeball) needed the same
acceleration, and TCG is slow enough that boot felt sluggish for
a moment. The boot is still under three seconds wall time, which
is fine. M2 onwards we'll feel TCG more ŌĆö every page-table walk
test runs through software emulation.

**Font provenance.** The TempleOS 8├Ś8 console font is the
preserved-as-public-domain artifact ZealOS keeps at
`src/Kernel/FontStd.ZC`. Verified the encoding by hand
(byte 0 = top row, bit 0 = leftmost pixel) before trusting the
blitter. The hex constants in `FontStd.ZC` match the layout in
the original TempleOS distribution; ZealOS just keeps the
literal data alive.

The serial line moved to print *after* all initialisation. Its

any of those fail, we triple-fault or panic before the sentinel
ever lands; CI catches both as a missing `FIELD_OS_BOOT_OK`.

## What M2 looks like

M2 is the memory manager. Per phase-0.md ┬¦M2:

- **PMM** ŌĆö bitmap allocator over the Limine memory map. One bit
  per 4 KiB frame, two-finger search for free frames.
- **VMM** ŌĆö page-table walker, `vmm_map(pml4, virt, phys, flags)`
  / `vmm_unmap` / `vmm_translate`. PML4 four-level walk; no
  recursive mapping (it's clever but it bites later under SMP).
- **Higher-half kernel** ŌĆö already mapped by Limine; keep
  PG_GLOBAL on kernel mappings so they survive CR3 reloads.
- **Per-process address spaces** ŌĆö fresh PML4s with the
  upper-half cloned from a master. Lower half is process-private.
- **Kernel heap** ŌĆö slab allocator with per-size caches at 16,
  32, 64, 128, 256, 512, 1024, 2048 bytes; large allocations
  fall through to a buddy/large-page path.

Exit criteria: 10,000 random-sized allocations / random-order
frees with zero leaks; map-and-unmap 1 GiB of test pages;
`vmm_translate` round-trips; smoke prints a free-RAM count.

Honest estimate per phase-0.md: 2ŌĆō3 FT-weeks, ~5 weeks part-time.
This is the milestone where design choices start to compound. I
do not expect M2 to compress the way M0 and M1 did.
